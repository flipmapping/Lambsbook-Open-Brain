# EXEC-DOCTRINE-009

## Title

Repository Convergence Doctrine

---

## Status

Founder Proposal

Recommended for Canonical Adoption

---

# Purpose

Establish the canonical execution doctrine governing repository convergence following the acceptance of bounded repository mutations.

This doctrine formalizes the distinction between executing a Repository Mutation Package and converging Repository Reality.

It applies to all repository-centric execution activities across the Lambsbook ecosystem.

---

# Fundamental Principle

Repository convergence is a repository evolution activity rather than a replay of historical mutation packages.

Once a bounded repository mutation has been accepted, Repository Reality becomes the Current Implementation Authority.

Subsequent execution SHALL converge the remaining repository toward Repository Reality.

---

# Repository Convergence Principle

Once any bounded portion of an approved Repository Mutation Package has been accepted into the Current Implementation Authority:

* Repository Reality becomes authoritative.
* Accepted implementation becomes the Current Implementation Authority.
* Historical mutation packages become implementation evidence only.
* Remaining implementation SHALL converge toward Repository Reality.

Execution SHALL NOT repeatedly replay superseded Repository Mutation Packages against an evolved repository unless explicitly authorized by the Founder.

---

# Repository Re-baselining Principle

Following every Founder-approved bounded repository mutation, Execution SHALL:

1. Inspect the updated Repository Reality.
2. Establish the updated Repository Reality as the Current Implementation Authority.
3. Discard superseded implementation assumptions.
4. Plan subsequent repository mutations only against the updated Repository Reality.

Repository convergence SHALL occur incrementally through verified bounded mutations.

---

# Historical Mutation Principle

Repository Mutation Packages record approved implementation intent.

They do not remain the implementation authority after repository convergence.

Following acceptance, Repository Mutation Packages become historical implementation evidence.

Repository Reality supersedes historical implementation packages.

---

# Repository Planning Principle

Future repository mutations SHALL be planned relative to the Current Implementation Authority rather than historical implementation packages.

Repository inspection SHALL precede implementation planning.

Execution SHALL verify Repository Reality before preparing subsequent repository mutations.

# Relationship to Repository Reality

Repository Reality remains the sole authoritative representation of implemented state.

Repository convergence updates Repository Reality through bounded accepted mutations.

---

# Relationship to EXEC-STD-001

EXEC-STD-001 governs execution behaviour.

EXEC-DOCTRINE-009 governs how execution progresses following repository acceptance.

The doctrine complements EXEC-STD-001.

---

# Relationship to Repository Mutation Packages

Repository Mutation Packages authorize bounded implementation work.

They SHALL NOT be repeatedly replayed after Repository Reality has evolved.

Subsequent Repository Mutation Packages SHALL be generated relative to the updated Current Implementation Authority.

---

# Lessons Learned

This doctrine originates from repository convergence experience gained during bounded mutation work packages where repeated replay of historical mutation packages introduced unnecessary revision cycles.

Repository convergence is simplified when execution continuously re-baselines against Repository Reality.
