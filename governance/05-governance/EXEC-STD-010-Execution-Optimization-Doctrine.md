# EXEC-STD-010 — Execution Optimization Doctrine

## Status

Founder Approved

Canonical Standard

---

## Purpose

Execution conversations exist to deliver production software.

Execution governance, standards, frameworks and tooling exist only to improve verified repository delivery.

---

## Optimization Principle

Working software takes precedence over execution tooling.

Execution tooling SHALL evolve only when:

- an objectively verified execution blocker exists;
- the blocker prevents a Founder-approved Repository Mutation Package; or
- the Founder explicitly authorizes framework evolution.

Otherwise the simplest objectively proven implementation mechanism SHALL be used.

---

## Artifact Priority Rule

When Repository Truth is sufficient and implementation authority has been granted, the primary execution deliverable SHALL be the complete executable Python Repository Mutation Package.

Implementation instructions SHALL NOT replace the executable artifact.

---

## Completion Rule

Repository inspection,

Repository Truth,

Gap Analysis,

Implementation reasoning,

and mutation planning

exist solely to support generation of the Repository Mutation Package.

Execution SHALL continue automatically until the complete standalone executable Python Repository Mutation Package has been produced unless objectively blocked by insufficient Repository Truth.

---

## Implementation

Canonical governance only.

No runtime changes.
